Overview

BackupVerified is built to be simple: a lightweight agent creates MySQL backups, encrypts them locally, and uploads them to your own AWS S3 bucket. On a schedule, the verification system performs test restores in a clean environment to prove the backups are usable.

Requirements

Before installing, make sure you have the following ready.

AWS S3 bucket

Create a bucket in your AWS account to store encrypted backups.

Server access

SSH access to the machine running MySQL and permission to run scheduled tasks.

Backup user

A MySQL user with minimum required privileges to dump the database.

Encryption key

A secure key/passphrase for encrypting backups before upload.

Security reminder

Do not paste passwords, private keys, or secrets into support emails. Keep secrets in your environment variables or secure configuration.

Install the agent

Download the BackupVerified agent and place the configuration file on your server.

Install (example)

CLI 
curl -fsSL https://backupverified.com/downloads/agent-latest -o backupverified-agent
chmod +x backupverified-agent
sudo mv backupverified-agent /usr/local/bin/backupverified

Where to put config

A common path is:

/etc/backupverified/config.yml

Configuration

Fill in your MySQL connection details, S3 destination, and schedule. Keep secrets out of your repo and prefer environment variables where possible.

config.yml (example)

YAML 
mysql:
  host: "127.0.0.1"
  port: 3306
  user: "backup_user"
  database: "app_db"

backup:
  schedule: "0 */6 * * *"
  retention_days: 14

encryption:
  key_env: "BACKUPVERIFIED_ENCRYPTION_KEY"

storage:
  provider: "s3"
  bucket: "your-s3-bucket-name"
  region: "us-east-1"
  prefix: "mysql-backups/app_db"

Schedule

Use cron format to control when backups run.

Retention

Control how long backup files are kept in your S3 bucket.

Run your first backup

After configuration, run the agent once to confirm it can create and upload an encrypted backup.

Run

CLI 
export BACKUPVERIFIED_ENCRYPTION_KEY="your-strong-key"
backupverified run --config /etc/backupverified/config.yml

What to check

  • A new encrypted object appears in your S3 bucket under the configured prefix.
  • The agent reports a successful upload and exits cleanly.
  • The portal (once enabled) shows a successful backup event.

Restore verification

Verification runs periodically download your encrypted backup and perform a test restore in a temporary environment. Results are recorded so you have proof your backups work.

What verification checks

A restore completes successfully and the database is readable after import.

How often it runs

Frequency depends on your plan and schedule (nightly, weekly, etc.).

Portal & alerts

The client portal provides visibility into backup history, verification results, and basic health signals. Alerts and notifications can be configured as available in your account.

Recommended alerting

Notify on failed backups, failed verification restores, and missing scheduled runs.

Troubleshooting

Backup fails to upload to S3

Check IAM permissions for PutObject, verify bucket region, and confirm network connectivity from the server.

Backup completes, but verification fails

Common causes include corrupted dumps, insufficient disk space in the restore environment, or incompatible MySQL versions. Review verification logs and adjust settings.

Scheduled runs not firing

Confirm cron/service scheduler is configured correctly and that the agent has execution permissions.

FAQ

Backups are encrypted before leaving your server. This helps reduce risk during transfer and storage.
You create and control the S3 bucket. BackupVerified uploads encrypted backup objects into your account.
Verification runs restore the backup in a clean temporary environment and records the result in your account history.
